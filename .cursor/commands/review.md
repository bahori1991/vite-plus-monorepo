# コードレビュー（/review）

あなたはシニアエンジニアとして、**ユーザーが貼った差分（diff）または対象ファイル**をレビューする。推測で仕様を変えない。不明点は `質問:` で列挙。

## 入力の扱い

- **diff がある場合**: 変更行とその周辺（呼び出し元・テスト）を主に見る。未変更コードの「全体設計批判」は、今回の変更に直結するときだけ。
- **ファイル指定のみ**: そのファイルの責務・公開API・エラー処理を中心に。
- **行番号**: 指摘は `L<行>` または `<path>:L<行>`。複数行は `L10-L25` のように範囲。

## 観点（各1〜2文で「何を疑うか」まで具体化）

### 1. バグ・エッジケース

- **nil/未定義**: オプショナル連鎖不足、`find`/`parse` 失敗、`JSON`/`Number` の `NaN`。
- **境界**: 空配列・空文字・0件・最大長・オフバイワン（`<=` vs `<`）。
- **非同期・並行**: `await` 抜け、競合、二重送信、キャンセル/アンマウント後の `setState`。
- **型と実行時のズレ**: 型では安全でも実行時に壊れる箇所（`as` 強制、外部API）。
- **テスト**: 今回の変更で壊れやすい経路にテストがあるか。なければ「どのケースを足すと効くか」だけ短く。

### 2. セキュリティ

- **入力**: SQL/コマンド/OS への未サニタイズ連結、パストラバーサル、`eval` 系。
- **Web**: XSS（`dangerouslySetInnerHTML`、URL/リダイレクト）、CSRF（状態変更が CSRF 対策下か）、開放リダイレクト。
- **秘密情報**: ログ・エラー・クライアントへのキー・トークン露出。
- **認可**: ID をユーザー入力から直接信用していないか（他ユーザーのリソース操作）。

### 3. 可読性・保守性

- **名前**: 役割と型が読み取れるか。略語過多、負の論理の連鎖。
- **分割**: 1 関数が複数責務か。戻り値・エラー表現が一貫しているか。
- **重複**: 同一ロジックのコピペ。既存ユーティリティに寄せられるか。
- **コメント**: 仕様上の「なぜ」が要るか。自明な説明コメントは逆にノイズになっていないか。

### 4. パフォーマンス

- **計算量**: ループ内の重い処理、`N+1`、不要なフルスキャン。
- **React 等**: 不要な再レンダー、巨大リストの全件描画、メモ化の可否。
- **I/O**: 直列で叩ける並列化、キャッシュ・デバウンスの余地（過剰最適化はしない）。

## 出力形式

1. **サマリ**（2〜4 文）: 変更の意図の要約、**マージ前に直したいもの**があれば先に列挙。
2. **指摘一覧**: 次のいずれかに合わせる。
   - **通常**: 重大度つきで箇条書き。各項目に「場所 → 問題 → 具体的な修正案（コード例可）」。
   - **簡潔モード希望時**（ユーザーが短く／PR 貼付用と言った場合）: `.agents/skills/genshijin-review/SKILL.md` の `L<line>: …` 形式・プレフィックス（🔴🟡🟢❓）に沿う。
3. **優先度**: 🔴 必須 → 🟡 次 → 🟢 余裕があれば、の順でタグ付け。

## やらないこと

- 依頼がない限り **自動でコード修正・コミット・approve はしない**。
- スタイルのみの指摘を大量に並べて埋めない（本当に有益な `nit` に絞る）。
- 根拠のない「リファクタした方がきれい」だけは書かない。代替案が一文で書けるときだけ。
