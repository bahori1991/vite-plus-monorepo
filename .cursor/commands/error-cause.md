# エラー原因の確認（/error-cause）

あなたはデバッグ担当として、**ユーザーが貼ったエラー（ログ・スタックトレース・再現手順）**から原因を特定する。推測だけで断定しない。不足情報は `質問:` で列挙。

## タスク

1. **エラーの種類**を分類する（ビルド／型／ランタイム／テスト／CI／DB／ネットワークなど）。
2. **最初に失敗した箇所**（根本原因に近いフレーム）を特定する。ラップされた `Caused by` / `error.cause` も辿る。
3. **再現条件**（コマンド・環境・入力）を明文化する。
4. **修正案**は「原因が確定した範囲」だけ。未確定なら検証手順を先に出す。

## 調査で使うコマンド（必要に応じて実行）

リポジトリ直下をカレントにし、状況に合わせて選ぶ。

### 直近の変更・履歴

```bash
git status
git diff
git log -5 --oneline
```

### 文字列・シンボルの所在

```bash
rg -n "エラーメッセージの断片|関数名|ファイル名" .
```

### モノレポ全体の健全性（ルート `package.json` のスクリプト）

```bash
pnpm run ready
```

個別パッケージなら（例）:

```bash
pnpm exec vp run test --filter <パッケージ名>
pnpm exec vp run build --filter <パッケージ名>
```

### Node / TypeScript のスタックを読むとき

- **自前コードのフレーム**（`packages/` / `apps/` 配下）を優先して辿る。
- **`node_modules` だけ**なら、どの依存のどの関数かを特定してから、呼び出し元（プロジェクト側）に戻る。

### `error.cause` がある場合（ES2022）

- 外側のメッセージだけでなく、**内側の `cause` チェーン**まで見る。

## 出力形式

1. **結論（1〜3文）**: 何が原因か。未確定なら「最有力」と「確定に必要な情報」。
2. **根拠**: スタックの該当行・ログの該当行・関連コードの参照（可能なら `path:L行`）。
3. **次のアクション**: 再現コマンド、追加ログ、仮説を潰す1ステップ。
4. **質問:**（任意）入力・環境・期待動作が足りないときだけ。

## やらないこと

- ログにない **憶測で仕様やデータを捏造**しない。
- 原因が曖昧なのに **大規模リファクタ**を提案しない。
- ユーザーが依頼しない限り **勝手にコミット・デプロイ**しない。
